Q_rad explorer — visual test checklist
=======================================

Run this in a browser where the app is reachable:
  - local:  `make start` in the repo, then open  http://localhost:8771
  - VPS:    https://tausorting.mihac.de   (float32 build already deployed)

The first load precomputes (~10-30 s locally) before the Q_rad tab draws. The layout:
left column = controls, right column = a plot panel with two tabs (Q_rad / Atmosphere).
The Q_rad tab has three stacked panels: (top) the TAU-vs-LAMBDA binning diagram, (middle)
Q/rho, (bottom) residual vs the full-ODF reference.

Backend behavior for each step below has already been verified via the API; this checklist
is the visual confirmation of the frontend wiring.


1. TAU-GROUP COUNT + SIMPLE OPTIMIZE  ->  reflected on the tau-vs-lambda plot
---------------------------------------------------------------------------
The binning diagram (top panel) plots every wavelength sub-bin at (log10 lambda,
-log10 tau_Ros), colored by which (lambda, tau) group it lands in, with white group
boxes drawn on top. It must redraw immediately whenever the grouping changes.

  a. In "Optimize tau edges", set the "to N tau groups" box to 3, click "Optimize ->".
     EXPECT: the tau-edges text box fills with 4 edges (3 groups); the top diagram
     redraws with 3 tau-bands x 2 lambda cells = 6 group boxes, sub-bins recolored into
     6 groups. Metrics row shows "(lambda,tau) groups = 6", "bands = 18".

  b. Change the number to 6, click "Optimize ->" again.
     EXPECT: tau box now has 7 edges (6 groups); the diagram redraws with 12 group boxes
     (6 x 2), sub-bins recolored into 12 groups; metrics show groups = 12, bands = 36.

  c. Edit the tau-edges box by hand (e.g. remove an edge) and press Enter / click
     "Compute Q_rad".
     EXPECT: the diagram boxes + colors update to match the new edges every time.

  PASS = the top diagram's boxes AND sub-bin colors always match the current tau-group
  count with no stale state and no page reload needed.


2. ALL Q_rad OPTIMIZATION TOGGLES WORK AS INTENDED
--------------------------------------------------
Under "Optimize for Q_rad" are five checkboxes (tau, lambda, flags, grow N, per-group
lambda), a time budget (s), and an "Advanced optimizer settings" panel. Run a short
optimize (set the seconds box to ~30) after each change and watch the result.

  a. grow N off:   uncheck "grow N", Optimize -> EXPECT the tau-group COUNT does not change
     (it only nudges the existing edges); with grow N on it may add groups.
  b. tau / lambda / flags: unchecking a block freezes that axis (e.g. tau off = the tau
     edges don't move; lambda off = the lambda cut doesn't move).
  c. per-group lambda on (default): EXPECT after it finishes, each tau band shows its OWN
     lambda cut (or none) -- the vertical cut in the top diagram "jumps" per tau band, and
     a "per-group lambda active" line lists each group's cut. Re-running keeps refining the
     cuts (they should move further, not reset).
  d. Advanced panel:
       - metric: rms | max|Delta| | intQ  (intQ ignores the tau-window, per the hint)
       - method: coord-descent | Nelder-Mead
       - max tau-groups: capped at the host ceiling (8 locally; 5 on the VPS) -- you can
         ask for fewer, never more
       - min gap tau / min gap lambda: edge-spacing guards
     Change any of these, Optimize -> EXPECT it runs with no error and honors the setting.
       (An out-of-range value like an unknown metric is rejected server-side -> red status.)

  SCORING WINDOW (affects both Compute and Optimize):
  e. Set "Scoring window" to lo=-1, hi=2 and click "Compute Q_rad".
     EXPECT: the rms value CHANGES (narrowing to the photosphere raises it, ~1.2e8 vs the
     full-window ~6.5e7), and the middle+bottom plot panels show a faint ORANGE SHADED BAND
     spanning [-1, 2] marking where rms is measured. Widen back to -5..4 to clear it.

  STOPPING CONDITIONS (Advanced panel; 0 = that stop is off):
  f. target rms: set it just ABOVE the current rms (e.g. 7e7) and Optimize ->
     EXPECT: it stops almost immediately; final status reads "... stopped: target_rms".
  g. plateau evals: set e.g. 8, target rms 0, Optimize -> EXPECT it stops once rms stalls;
     status reads "... stopped: plateau".
  h. max evals: a small value (e.g. 30) -> status reads "... stopped: max_evals".
  i. time budget: a small seconds value -> "... stopped: max_seconds".
  j. "cancel" link during a run -> keeps the best binning found so far.

  PASS = every toggle/knob changes the run as described, and the final status names the
  stopping reason.


3. THE FINAL RESULT IS SHOWN AFTER FINDING AN OPTIMAL Q_rad
----------------------------------------------------------
  a. Run any Optimize for Q_rad (per-group lambda on, grow on, ~60 s).
     While running: the status ticks "optimizing Q_rad... N evals - rms ... - G groups - Ts".
  b. When it finishes, EXPECT:
       - status shows "Q_rad optimized: rms <start> -> <final> (-X%) - stopped: <reason> -
         N evals - computing..."
       - the tau-edges box + flags (or per-group lambda cuts) update to the optimized binning
       - the whole view recomputes: the top diagram redraws to the optimized grouping, the
         middle Q/rho and bottom residual curves update, and the metrics row (bands, groups,
         rms residual, max|dQ/rho|, intQ error, assigned sub-bins) reflects the final binning
       - "Download kappa table (.dat)" now exports that optimized table.

  PASS = the optimized binning is applied AND rendered (no need to click Compute again), and
  the metrics + plots match the reported final rms.


4. BONUS CHECKS (recent additions)
----------------------------------
  - MODEL dropdown + "Refresh": drop a file into models/ and hit Refresh; valid 4-column
    strictly-decreasing-z models appear in the dropdown, bad files are listed with the exact
    reason (wrong column count / non-numeric / z not strictly decreasing). Selecting a model
    reruns everything on it.
  - ATMOSPHERE tab: click it; EXPECT three panels (T, log10 rho, log10 P) vs depth-from-top
    [Mm] for the selected model, drawn essentially instantly (it reads the model .dat directly
    -- no ODF/precompute). Switch the model dropdown -> the atmosphere redraws for it.

- Every step above was verified at the data/API level; this file confirms the frontend renders
  and wires it correctly.
